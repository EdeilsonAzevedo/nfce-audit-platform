Dados de Origem

NF-e

Ponto de partida: base de compras públicas extraída das Notas Fiscais Eletrônicas (NF-e) do setor público estadual. Cada registro representa uma linha de item de compra.

Campos utilizados

Filtros de Formato

Notebook 01

Limpeza e validação estrutural dos GTINs brutos. Remove registros inválidos e isola o subconjunto de produto de interesse para o projeto.

✕ Remover registros nulos, zeros e descrições genéricas "SEM GTIN"
→ Manter somente EAN-13 (exatamente 13 dígitos, padrão GS1 Internacional)
→ Selecionar prefixo 789 — escopo GS1 Brasil exclusivamente
✓ Validar dígito verificador EAN (algoritmo módulo 10)
3

Score de Confiança por GTIN

Cada GTIN recebe um score de confiança calculado por subtração de penalidades. O score reflete a consistência dos dados declarados nas NF-e para aquele produto.

confiança = 1 − Σ penalidades
Critério Condição Penalidade
Dígito verificador Inválido −0.40
Múltiplos NCMs distintos n_ncm > 1 −0.30
Muitas descrições distintas n_desc > 5 −0.20
Preço muito variável CV > 1 −0.10
score ≥ 0.70 → Confiável score < 0.70 → Suspeito
4

Priorização por Pareto — Volume Financeiro

Notebook 02

Aplicação do princípio de Pareto (80/20) ao volume financeiro das compras. O objetivo é garantir máxima cobertura de impacto com o mínimo de GTINs a processar — foco nos produtos que mais pesam no orçamento público.

volume_item = valor_unit_liq × qtd_geral_item

GTINs são ranqueados em ordem decrescente de volume financeiro total e selecionados cumulativamente até atingir 95% do volume financeiro da base.

Clustering Semântico (DBSCAN)

As descrições de cada GTIN são agrupadas semanticamente usando embeddings de texto e DBSCAN. O objetivo é identificar a descrição dominante e detectar outliers que indicam possível confusão de produto.

Modelo de Embedding

MiniLM multilíngue

distância cosseno

Parâmetros DBSCAN

eps=0.25 min_samples=2

Roteamento por resultado

▶ Cluster dominante ≥ 70% → envia ao LLM para extração
⚠ Outliers ≥ 50% → sinaliza para revisão manual
→ Menos de 3 descrições → LLM direto (sem clustering)
7

Extração com LLM

Um Large Language Model (LLM) recebe a descrição normalizada do GTIN e retorna, em formato estruturado JSON, os atributos do produto. Suporte a dois modos: local (privacidade) ou cloud (qualidade).

Local — On-Premise

Gemma-3n-e4b

Sem custo por token Privacidade total dos dados Execução on-premise

Cloud — API

GPT-4o-mini

Alta qualidade de extração Custo por token (OpenAI) Menor latência de resposta
8

Atributos Extraídos por GTIN

Notebook 03

O LLM retorna um JSON estruturado com os atributos padronizados do produto. Cada campo é validado e o score de confiança é registrado junto ao motivo da classificação para rastreabilidade completa.

descricao Nome do produto normalizado e limpo
marca Fabricante ou marca do produto
volume Quantidade, peso ou capacidade unitária
ncm Código NCM sugerido pelo LLM (8 dígitos)
confianca Score de confiança + motivo textual (rastreável)
Saída persistida em gtin_resolved.json

/explorador Consulta e edição inline por GTIN — salva diretamente no JSON
